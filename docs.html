Overview

What is SheetQA?

A multi-agent AI pipeline that accepts a natural language query and a CSV/Excel file, autonomously plans an analysis strategy, writes Python code, executes it in a secure sandbox, validates the output, self-corrects up to 3 times, and returns the final answer.

📤

Upload & Ask

User uploads a CSV/Excel file and asks a question in plain English.

🧠

AI Plans & Codes

The LLM creates an analysis plan, then writes Python code to execute it.

🛡️

Sandbox Execution

Code runs in RestrictedPython — no file system, no imports, no network access.

🔄

Self-Correction

Validator Agent checks the output. On mismatch, loops back and re-plans (up to 3x).

API Reference

REST Endpoints

The FastAPI backend exposes three endpoints. The server runs on http://localhost:8000 with auto-reload.

POST /analyze Analyze a spreadsheet with a natural language query

Accepts a multipart form with a spreadsheet file and a query string. Runs the full LangGraph pipeline (validation → planning → execution → verification) and returns the result. Files are automatically cleaned up after processing.

Request Parameters

NameTypeRequiredDescription
file File required CSV, XLSX, or XLS file. Max 10MB, 50K rows.
query string required Natural language question about the data.

Success Response 200 OK

JSON { "status": "success", "final_answer": "[{'Region': 'East', 'Revenue': 30500}, ...]", "retry_count": 0, "validation_errors": [] }

Rejection Response 200 OK (status: rejected)

JSON { "status": "rejected", "final_answer": "File validation failed: Unsupported format '.txt'", "retry_count": 0, "validation_errors": ["Unsupported file format: '.txt'"] }

Status Values

success — query answered rejected — file invalid failed — max retries exceeded
GET /history Retrieve the last 20 analysis runs

Returns the most recent 20 runs from the SQLite database, ordered newest-first. Each entry includes the query, filename, status, retry count, and final answer.

Pipeline

How Each Node Works

The LangGraph pipeline has 5 nodes connected by conditional edges. Here's exactly what each node does.

1
file_validator
Pure Python validation — no LLM cost. Runs 5 sequential checks on the uploaded file. On any failure, sets status = "rejected" and short-circuits to the rejection handler.
File exists & not empty Extension is .csv / .xlsx / .xls Size < 10 MB Parseable by Pandas Row count < 50,000
2
rejection_handler
Terminal node. Formats the validation errors into a human-readable message and sets final_answer. The graph ends here — no LLM is ever called for invalid files.
3
planning_agent — LLM
Calls Groq Llama 3.3 with the user's question and DataFrame schema (columns, dtypes, first 3 rows, shape). Produces a numbered, step-by-step analysis plan. On retries, includes the previous code, execution error, and validator feedback to avoid repeating mistakes.
System Prompt You are a senior data analyst. Given a user question and a DataFrame schema, produce a precise, step-by-step analysis plan in plain English. Be specific about which columns to use, what calculations to perform, and what the output format should be. Do not write code.
4
execution_agent — LLM + Sandbox
Calls Groq to generate Python code based on the plan. Extracts code from markdown fences if present. Executes via RestrictedPython with df (DataFrame), pd (pandas), and np (numpy) injected. The code must assign its answer to RESULT. On execution error, increments retry_count.
No import No os / subprocess No __import__ No print() RESULT must be str/num/dict/list
5
validator_agent — LLM
Calls Groq to verify the execution output answers the user's original question. Returns a JSON verdict: {"verdict": "match"|"mismatch", "reason": "..."}. On match → sets final_answer and ends. On mismatch → increments retry and loops back to planning agent. After 3 failed attempts, ends with a graceful failure message.

Edge Routing Logic

route_after_validation()
status == "rejected" → rejection_handler
else → planning_agent
route_after_execution()
exec_error + retries ≤ 2 → planning_agent
exec_error + retries > 2 → END
success → validator_agent
route_after_validation_check()
verdict == "match" → END
mismatch + retries ≤ 2 → planning_agent
mismatch + retries > 2 → END
State Schema

AgentState TypedDict

The shared state flows through every node in the pipeline. Each node reads and writes specific fields. Defined in graph/state.py.

user_query
str
Natural language question from the user.
file_path
str
Absolute path to the saved upload file.
file_df
Optional[DataFrame]
Parsed pandas DataFrame. Set by file_validator on success.
validation_errors
list[str]
Validation failure reasons. Empty if file passed.
plan
str
Step-by-step analysis plan from the planning agent.
generated_code
str
Python code written by the execution agent.
exec_output
Optional[str]
The RESULT value from sandbox execution.
exec_error
Optional[str]
Error message if sandbox execution failed.
verdict
str
"match" or "mismatch" from the validator agent.
retry_count
int
Number of re-plan attempts. Ceiling is 3.
final_answer
str
The answer returned to the user.
status
str
"pending" | "validated" | "rejected" | "success" | "failed"
